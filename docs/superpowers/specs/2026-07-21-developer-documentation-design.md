# My_Agent 开发者文档设计

## 目标

为 My_Agent 建立一套面向开发者的中文文档。文档既要帮助新参与者完成构建和运行，也要帮助项目维护者从总体架构逐步深入到主要模块、核心类型和关键函数。

本次文档只描述当前 `main` 分支中已经实现且能够通过源码、命令或测试验证的行为。未来规划和尚未实现的 commandPolicy 通配符配置不属于本次范围。

## 目标读者

- 希望理解项目整体设计和运行机制的开发者。
- 准备修改 Agent Loop、模型访问、命令执行、Session 或 TUI 的维护者。
- 需要在本地构建、配置、调试 My_Agent 的贡献者。

读者应具备基本的 C++、CMake、终端和 HTTP API 使用经验；文档不负责讲解这些技术的通用基础知识。

## 文档结构

采用“README 入口 + 专题文档”的分层结构：

| 文件 | 内容 |
| --- | --- |
| `README.md` | 项目定位、主要能力、快速构建和运行、文档导航 |
| `docs/getting-started.md` | 依赖、环境变量、配置、构建、安装和首次运行 |
| `docs/architecture.md` | 总体分层、依赖方向、线程模型和核心数据流 |
| `docs/agent-loop.md` | 模型请求、回复解析、命令审批、Shell 执行、完成和停止流程 |
| `docs/modules.md` | 逐模块说明核心类型、关键函数、依赖、生命周期和典型调用路径 |
| `docs/configuration.md` | `.env`、`agent.yaml`、CLI 参数及配置优先级 |
| `docs/session-and-tui.md` | Session 持久化、TUI 状态、事件、日志和交互命令 |
| `docs/troubleshooting.md` | 构建、配置、Provider、终端和数据库常见问题 |

现有的 Session、TUI 和 commandPolicy 专题资料继续保留。新文档负责建立统一入口并整合仍然有效的信息，通过链接指向需要保留的深入内容，避免复制出多份可能互相冲突的行为说明。

## 内容组织原则

### 学习路径

开发者从 README 开始，依次阅读快速开始、总体架构、Agent Loop、模块导读，再按需阅读配置、Session/TUI 和故障排查。README 保持简洁，不承载实现细节。

### 模块说明模板

`docs/modules.md` 中的每个模块统一包含：

1. 模块职责与边界。
2. 主要头文件和实现文件。
3. 核心类型、关键字段及生命周期。
4. 关键函数的输入、输出、异常和副作用。
5. 上游调用者与下游依赖。
6. 一条典型调用路径。
7. 对应测试文件的简要索引。

模块导读重点解释代码的设计意图和协作关系，不逐行复述实现。主要讲解以下主链路：

```text
main
  -> SessionManager
  -> AgentSession
  -> Agent Loop
  -> ModelClient / CommandAuthorization / Shell
  -> AgentEvent / AgentRunResult
  -> Console 或 TUI
```

### 图示与源码导航

复杂调用关系使用 Mermaid 图展示，包括总体依赖、Agent Loop 时序和 TUI Worker 数据流。文档使用仓库内相对链接指向源码和其他文档，使读者能够从说明直接定位实现。

## 准确性边界

- 当前源码是行为说明的唯一事实来源；现有设计文档只能作为线索。
- 文档中的构建、安装、测试和 CLI 命令需要实际执行或通过等价的只读方式验证。
- 明确记录当前限制，包括 Shell 解析边界、协作式停止、输出截断、配置查找路径和命令审批边界。
- 不写入 API key、个人环境值或其他敏感信息。
- 不承诺未实现功能，不把待办事项写成当前能力。
- 同一事实指定一个主要维护位置，其他文档通过链接引用。

## 错误处理文档

故障排查按错误来源组织：

- `.env` 或 `agent.yaml` 缺失、字段缺失及格式错误。
- HTTP、Provider 和模型响应协议错误。
- 命令被策略、人工审核或缺少审核能力拒绝。
- Shell 子进程失败和输出截断。
- Session 数据库创建、读取、恢复和容量限制错误。
- TUI 终端兼容性及退出、停止相关问题。

每类问题说明可观察现象、常见原因和安全的检查方法。只有源码中存在明确恢复方式时，才提供修复步骤。

## 测试内容范围

测试内容保持精简：

- README 提供构建和运行测试的基本命令。
- 模块文档只列出相关测试文件，不逐项讲解测试用例。
- 不创建独立测试专题文档。

尽管文档中的测试讲解简短，交付前仍需运行完整测试套件，以验证文档描述与当前实现一致。

## 验收标准

- 新开发者能够仅依赖文档完成配置、构建、启动 TUI 和执行 Console 任务。
- 读者能够从入口文档定位任一主要模块及其核心接口。
- Agent Loop、命令授权、Session 持久化和 TUI 线程模型具有清晰的调用或数据流说明。
- 所有仓库内链接有效，示例命令与当前 CLI 和配置格式一致。
- 文档只描述当前实现，不包含未实现的通配符策略等规划内容。
- README 简洁，专题文档之间没有明显冲突或大段重复。
- 项目能够成功构建，完整测试套件通过。

## 非目标

- 不生成逐个符号的 Doxygen HTML 站点。
- 不逐行解释所有实现文件。
- 不编写完整测试用例参考手册。
- 不设计或实现新功能。
- 不修改现有运行行为、公共接口或配置格式。
